## Decisions made and why

### 1. V1 uses Supabase with anonymous auth — not localStorage
**Decision:** Persistence moves to a Supabase backend. `signInAnonymously()` creates a stable per-device `user_id` with no sign-up friction.
**Why:** The professor requires a database connection. The "no accounts" experience is preserved — anonymous auth means zero friction — while storage becomes durable, server-side, and RLS-isolated per user. The localStorage path was already named as a temporary measure in the original brief; this is the natural next step it anticipated.
**Trade-off accepted:** A user on a new device starts fresh until accounts proper are introduced. That's fine for V1.
**Where it lives:** [project-context.md](project-context.md) §6 and the persistence note.

### 2. V1 is a browse-first catalog, not a conversational agent
**Decision:** Chat, Claude research, and recommendation cards are cut from V1 and documented as Phase 2 (V2). V1 is locked to five features: browse by category, search, product detail, clean evaluation, save to a personal list.
**Why:** The professor's recommended scope is a tight catalog. One-week timebox makes the conversational agent — which has the most technical unknowns (live web search, ingredient data, prompt design) — a risk to the whole build. Cutting it doesn't delete it; the backend built in V1 is exactly what the agent needs in V2.
**Where it lives:** [feature-plan.md](feature-plan.md) Phase 1 (V1) and Phase 2 (V2).

### 3. Clean evaluation is the 5th feature — not a "nice to have"
**Decision:** The professor's list of four was adopted in full, with clean evaluation added as the fifth.
**Why:** Without the `clean_score` + ingredient breakdown surfaced in the catalog, V1 is a generic shopping list with no reason to exist. It's already in the schema (`clean_score` cached on `products`, `product_ingredients`, `concern_level`/`concern_reason`). It's zero additional infrastructure; it's just rendered.
**Where it lives:** [feature-plan.md](feature-plan.md) Feature 4.

### 4. The "clean standard" is now concretely defined for V1
**Decision:** Three-layer rule — (1) EWG Skin Deep as primary source, regulatory banned lists as hard floor; (2) `clean_score` from ingredient concern levels weighted by `position` (label prominence); (3) most conservative source wins on conflict, source always surfaced. V1 displays and explains the stored `clean_score` — live recomputation is out of scope.
**Why:** Without a concrete definition, the clean evaluation can't be consistent or explainable, and trust breaks on the first wrong answer. The schema already implements this model; the decision is what it's implementing *against*.
**Where it lives:** [project-context.md](project-context.md) §7, *Clean standard (V1 definition)*.

### 5. V1 deadline is 2026-06-20/21 (one week)
**Where it lives:** [project-context.md](project-context.md) §6 deadline note, [build-plan.md](build-plan.md) intro.
